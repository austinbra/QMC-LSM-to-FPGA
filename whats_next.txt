# ✅ Project Context – QMC‑LSMC American Option Pricing FPGA Pipeline  
(Use this in ALL prompts for this project!)

---

## 🎯 Project Scope / Goal  
- Build a **production‑grade QMC‑LSMC pipeline** for **American option pricing** on a Spartan‑7 FPGA using Vivado.  
- Pipeline stages:  
  **Sobol** (QMC generator) → **InvCDF** (inverse normal) → **GBM** (path simulation) → **Accumulator** (regression sums, batched) → **Regression** (β solve via Gaussian elimination) → **LSM Decision** (exercise logic) → **UART / output**.  
- Must be **robust, efficient, parameterized, simple, and correct**.  
- Clock target: ~170 MHz on Spartan‑7.  
- Accuracy requirement: <1% relative error (errors bounded by QMC variance).  
- Style: Low‑level, clear SystemVerilog RTL with ready/valid handshaking and stall safety.  

---

## 🔑 Critical Design Principles
- **Most important:** “Make sure everything is completely correct or I will be sad.”  
- All code must **survive stalls, retiming, replication, and batching without data loss or corruption**.  
- **Handshakes**:
  - Use **ready/valid** protocol correctly at every stage.  
  - Include **skid buffers** (one‑deep elastic registers) for inputs.  
  - For branching logic, use explicit **barrier signals** to prevent desync.  
  - Valid and ready must never glitch or race.  
  - Assertions required to enforce handshake invariants.  
- **Parameterization**: No magic numbers. Widths, QINT, QFRAC, LATENCIES, LUT sizes, N_SAMPLES, NUM_LANES must come from `fpga_cfg_pkg`.  
- **Declarations**: *Never declare a variable after it is used.* All declarations must appear before usage in procedural code.  
- **Reset conventions**: Use `posedge clk or negedge rst_n` with `'0` initialization. Avoid implicit latches.  
- **Pipeline invariants**: Always align valid/data by construction. Insert registers if needed for alignment.  
- **Mathematical behaviors**:  
  - fxMul: DSP‑mapped, scaled fixed‑point, pipeline‑latency param.  
  - fxDiv: Wrapper of `div_gen` IP, saturating divide‑by‑zero safe.  
  - fxExpLUT, fxlnLUT: BRAM lookup, range clamping, stall‑safe.  
  - fxSqrt: LUT + Newton–Raphson iterations, fully pipelined.  
  - InvCDF: Zelen–Severo rational approximation, with correct negate pipelining.  
  - Accumulator: 64‑bit wide sums, saturate down to WIDTH, state machine triggers regression at batch end.  
  - Regression: Deep pipeline Gaussian elimination, pivot+normalize+elim stages, sticky singular_err, fallback to mean payoff.  
- **Batching**: Use fixed‑size “mini‑batches” of paths; accumulate regression sums across batches; at the end run regression once → β’s → final PV sweep/average.  

---

## 💻 Implementation Best Practices
1. **Resource fit**: Use DSPs for mul/div, BRAM for LUTs (Sobol directions, log/exp LUTs). Spartan‑7 resource budget must be respected (≈80 DSP, 60–70 BRAM typical).  
2. **Assertions**: For handshake, stall stability, overflow stickiness.  
3. **Debugging**: Add `$display` in testbenches; check waves for handshake signals and regression matrix states.  
4. **Mini‑batches**: Clear only per‑batch storage (GBM outputs), preserve accumulators until end of all paths.  
5. **Testbenches**: Exercise stall behavior, backpressure, edge cases (zero input, max/min values). Compare against **quantized C++/Python golden model**.  
6. **Top‑level / lanes**: Lane replication implemented in top‑level with `generate`. Merges per‑lane accumulators if needed. UART used for Python/C++ hybrid demo with real market data.  
7. Do not use SystemVerilog reserved keywords (like expect, unique, priority, etc.) as variable, signal, or task/function parameter names. Rename them (e.g. use exp_v instead of expect). Always declare variables at the top of a block/task/function before procedural statements (for Vivado/older parsers that don’t support mixed declaration/statement).

## 🚨 Required Style Rules
- **Correctness first.** Everything must compile, synthesize, and be logically/stall‑correct.  
- **No declarations after use.** EVER.  
- **valid_out must never assert when ready_in = 0 unless result/data hold stable.**  
- **Backpressure must never overwrite registers.**  
- **Barrier readiness required in branching logic.**  
- Always saturate or clamp fixed‑point math where overflow could occur (e.g., accumulators, exp_arg).  

---

▶ Shift Operator Semantics (SystemVerilog)

- >> : Arithmetic right shift if operand is signed (sign bit replicated). Unsigned → logical right shift. Use when you want division by powers of two that preserves sign for signed operands.

- >>> : Explicit arithmetic right shift. Always performs sign‑extension regardless of context. Use this when you want to be 100% sure negative values shift correctly (e.g., sigma2 >>> 1).

- << : Logical left shift. Zeros shifted in. Common for scaling unsigned values or when sign is irrelevant.

- <<< : Explicit arithmetic left shift. Shifts left but preserves signedness semantics. Use for fixed‑point multiply by power‑of‑two (e.g., (1 <<< QFRAC) for 1.0 scaling).

⚠️ Rule of thumb:

- Use >>> for safe arithmetic right shift.

- Use <<< for safe arithmetic left shift in fixed‑point contexts.

- Do not rely on implicit type casting. Always cast with $signed(...) where sign matters.

- Assert correctness when shifting signed values to catch overflow/extension issues.


---

🧾 Best Practice Reminders

- Always use &&/|| for logical conditions (never &/| unless operating bitwise).

- Don’t reuse | where you mean short‑circuit OR.

- Don’t let one branch run if the other is stalled; use barrier_ready.

- Skid buffers should be one‑deep, reset to zero.

- All operands declared before they’re used.

- Saturate accumulators and intermediate values when reducing width.

- Keep state machines with explicit one‑hot or small enums; clear outputs on reset and transitions.

- Assertions everywhere:
	- Output stable under stalls.

	- Singular error flags sticky.

	- Inputs not overwritten under backpressure.


- Always test with randomized stalls on ready_in.


---

💻 Implementation Best Practices

1. Use DSP48 (fxMul/fxDiv) and BRAM (LUTs, Sobol direction) primitives efficiently.

2. Simplify loops by unrolling when it improves synthesis/readability.

3. One‑deep skid buffers for handshake compliance.

4. Consistent naming: skid_valid, barrier_ready, shift_en.

5. Assertions checked inside simulation always.

6. Keep code lane‑friendly: NUM_LANES param supported in top‑level generate.

7. Reset accumulators at correct time (per‑batch vs per‑simulation total).
